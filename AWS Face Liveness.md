# Laveyoo AWS Face Liveness

**Plugin details**
Perform highly secure biometric identity verification by integrating an interactive Facial Liveness check into your application.
This includes detecting spoofing attempts (e.g., holding up a printed photo or iPad) and ensuring the user is a live, physically present human being.

The following element is provided:
- AWS REKOGNITION - FACE LIVENESS (FRONT-END WEB COMPONENT)

The following actions are provided:
- START LIVENESS SESSION (BACK-END)
- GET LIVENESS RESULTS (BACK-END)

The plugin returns an interactive camera UI for the end-user. Upon completion, it returns a confidence score (0-100), a definitive `is_live` boolean, and a high-quality `reference_image` (Base64) captured during the session.

## 📖 Instructions

1️⃣: AWS REKOGNITION - FACE LIVENESS (FRONT-END WEB COMPONENT & BACK-END ACTIONS)
================================================================================

📋ELEMENT DESCRIPTION
--------------------------------
The AWS Face Liveness Web Component provides a secure, interactive camera interface that prompts users to move their face into an oval on-screen. Video is streamed securely to AWS Rekognition to determine if the user is a live, physically present human being, preventing presentation attacks (spoofs).
This front-end element is built as a highly optimized, native Web Component (`<laveyoo-face-liveness>`), ensuring it loads instantly and securely within any Bubble.io app without conflicting with other libraries.

🔧 STEP-BY-STEP SETUP
--------------------------------
0) Sign-up for an AWS Account and access the IAM console: https://console.aws.amazon.com/iam/home

1) Create your AWS REKOGNITION ACCESS KEY & SECRET ACCESS KEY. Create an IAM user and attach a policy allowing the `rekognition:CreateFaceLivenessSession` and `rekognition:GetFaceLivenessSessionResults` actions.

2) Enter in the **Shared** tab of your Bubble PLUGIN SETTINGS the `AWS Access Key`, `AWS Secret Key`, and the `AWS Region` (e.g., `us-east-1` or `us-west-2`). 

3) Create the Workflow to start the session. Before showing the camera, you must use the **START LIVENESS SESSION** backend action to generate a unique `SessionId` from AWS.

4) Add the **Face Liveness Camera** element to your page. Set it to be hidden on page load. Once you have the `SessionId` from step 3, show the camera element and pass the `SessionId` directly into its properties via a Custom State.

5) Integrate the logic into your application using the following element's states and events:

FIELDS: 
- SESSION ID: The unique ID generated by the Backend Start Session Action.
- REGION: Your AWS region (e.g., us-east-1).

EVENTS:
- CHECK COMPLETED: Event triggered when the camera UI successfully finishes capturing video and AWS begins analyzing it.
- ERROR OCCURRED: Event triggered upon an error (e.g., camera access denied, or network failure).
- IS NOT ACTIVE: Event triggered if the component unmounts unexpectedly.

EXPOSED STATES:
- ERROR_MESSAGE: Populated upon ERROR OCCURRED event.

ELEMENT ACTIONS - TRIGGERED IN WORKFLOW:
- GET LIVENESS RESULTS (BACK-END): Once the `CHECK COMPLETED` event fires on the front-end element, you must run this secure server-side action to fetch the actual score.

   Inputs Fields:
      - SESSION ID: The same ID used to start the session.
   
   Output Fields:
      - STATUS: Text indicating the final status (e.g., "SUCCEEDED", "FAILED").
      - CONFIDENCE SCORE: A number from 0-100 indicating the likelihood the user is live.
      - IS_LIVE: A Yes/No boolean stating if the confidence score meets a standard threshold.
      - REFERENCE_IMAGE_BASE64: A clean, cropped image of the user's face taken during the active check formatted as a Base64 string. 


🔍INTEGRATION WORKFLOW EXAMPLE
======================
1. User clicks "Verify". Backend runs **Start Liveness Session**.
2. **Face Liveness Camera** element opens and user performs the motion challenge.
3. Upon `CHECK COMPLETED` event, run **Get Liveness Results**.
4. If `is_live` is "Yes", proceed to next step (e.g. comparing the reference image with an ID).

⚠️TROUBLESHOOTING
================
Any plugin related error will be posted to the Logs tab, "Server logs" section of your App Editor.
Make sure that "Plugin server side logging" and "Plugin client side logging" are selected in "Show Advanced".

For front-end actions, you can also open your browser's developer console (F12 or Ctrl+Shift+I in most browsers) to view detailed error messages and logs.

Always check the ERROR MESSAGE state returned by backend actions and implement error handling on the front-end to provide a better user experience.

⚡PERFORMANCE CONSIDERATIONS
===========================

⏱️BACK-END ACTION START DELAY
-----------------------------------------------
Each time a server-side action is called, Bubble initializes a small virtual machine to execute the action. If the same action is called shortly after, the caching mechanism kicks in, resulting in faster execution on subsequent calls.

A useful workaround is to fire a dummy execution at page load, which pre-warms the Bubble engine for the next few minutes, reducing the impact of cold starts for your users when fetching Liveness scores.
